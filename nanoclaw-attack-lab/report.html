이 요청 자체는 정상 업무처럼 보인다. 공격 지시는 사용자 프롬프트가 아니라 MCP 도구 설명 안에 숨어 있었다.

각 성공은 safe_sink 도구 호출이 실제로 발생했음을 의미한다. 기록은 모두 로컬 canary이며 실제 secret이나 외부 시스템을 대상으로 하지 않았다.

해석

가장 약한 경계

Tool Metadata

tool description은 모델에게 “도구 사용법”으로 강하게 작동한다. 이 경계에서 숨은 지시가 도구 호출 권한으로 이어졌다.

2차 성공 조건

Workflow Framing

문서, 검색 결과, tool response는 처음엔 막혔지만 “local lab bookkeeping”, “pending callback”처럼 업무 완료 절차로 포장하자 실행됐다.

핵심 결론

Trust Boundary

NanoClaw/Codex 조합에서는 tool description뿐 아니라 retrieved content와 업무 레코드 필드가 tool call 권한으로 승격되지 않도록 provenance 정책이 필요하다.

방어 방안

  • MCP tool description을 실행 지시가 아니라 정적 메타데이터로 취급한다.
  • 외부 MCP 서버의 tool description diff를 기록하고 allowlist/pinning을 적용한다.
  • safe_sink 같은 민감 도구는 사용자 직접 요청 또는 harness 권한에서만 호출 가능하게 제한한다.
  • tool response, RAG 문서, 첨부 문서는 provenance tag를 붙이고 도구 호출 권한을 부여하지 않는다.
  • evidence 파일처럼 tool call 감사 로그를 남겨 재현 가능한 보안 평가 자료로 만든다.
